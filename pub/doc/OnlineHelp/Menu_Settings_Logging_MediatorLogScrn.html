View Mediator Transaction Logger Details Screen
Screen Heading: Settings > Logging > View Mediator Transaction Logger Details
Use this screen to view settings for the Mediator transaction logger.
To open the View Mediator Transaction Logger Details screen, click Logging under the Settings menu on the Navigation panel. From the logger list, click Mediator Transaction Logger.
Click Edit Mediator Transaction Logger to change the settings for the security logger.
Name
The name of this logger, Mediator Transaction Logger.
Enabled
Whether the logger is to write entries to the transaction log. The default is No.
Mode
Mode for writing log entries. The default is Synchronous.
Option
Description
Synchronous
The logger writes log entries directly to the destination.
You might use synchronous logging when you have an application that requires some type of auditing to succeed, and you do not want to proceed without knowing that the auditing occurred.
If synchronous logging for an entry fails (for example, because the destination is unavailable), the logger switches to asynchronous logging for that entry (see below).
Asynchronous
The logger writes log entries to a queue, and the queue later writes the entries to the destination.
Guaranteed
How Integration Server is to maintain the queue when the logger is logging asynchronously. The default is No.
Option
Description
No
Integration Server maintains the queue in memory. This option provides better logging performance. However, if Integration Server shuts down abnormally, the log entries in the queue will be lost.
Yes
Integration Server persists the queue on disk. This option is safer but can adversely affect logging performance.
Maximum Queue Size
For asynchronous logging, maximum number of log entries the queue can hold. The default is 100000. Specify numerals only; for example, do not include commas or periods.
Choose a value that accommodates your system's average volume for log entries. If your logging volume has sudden spikes, the queue can usually catch up by writing the pending entries during lulls. Make sure the Integration Server host machine has enough disk space or memory to accommodate the largest possible size of the queue as specified in this field, as well as the requirements of other applications the Integration Server is hosting. If the logger is writing to a database, the queue's insertion of logged data into the database is constrained by the database's availability and connections limit.
If the queue reaches its maximum, the logger writes the log entries to a file called FailedAuditRecords.log in the Integration Server_directory /instances/instance_name/logs directory. You can scan the file to find events that not logged.
Destination
Destination for log entries.
Option
Description
Database
The logger writes log entries to an external RDBMS. You must also set the Maximum Retries and Wait Between Retries fields.
This value is available when the IS Core Audit Log connects to an external RDBMS at Integration Server startup.
File
The logger writes log entries to file. This is the default.
Maximum Retries
If the destination is set to Database, number of times the logger (synchronous) or queue (asynchronous) should try to connect to the database. The default is 3 retries.
For synchronous logging, the logger tries to write each log entry up to the number of times specified, then switches to asynchronous logging.
For asynchronous logging, the queue tries to write each log entry up to the number of times specified, then writes the entry to a file called FailedAuditRecords.log in the Integration Server_directory /instances/instance_name/logs directory. You can scan the file to find events that not logged.
Wait Between Retries
If the destination is set to Database, number of seconds the logger should wait between tries to connect to the RDBMS. The default is 5 seconds.
