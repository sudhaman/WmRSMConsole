Edit HTTPS Diagnostic Port Configuration Screen
Screen Heading: Security > Ports > Edit HTTPS Port Configuration
Use the Edit HTTPS Diagnostic Port Configuration screen to view or specify the settings for the diagnostic port, a special port that uses threads from a dedicated thread pool to accept requests for services via HTTPS.
To open the Edit HTTPS Diagnostic Port Configuration screen, go to the Add Port Screen and select HTTPS Diagnostic. Click Submit.
Click Return to Ports to return to the list of ports that are currently defined on the server.
Diagnostic HTTPS Listener Configuration
Enable
Controls whether or not the port is enabled. When the port is enabled, it can receive requests.
Note:  
A disabled port is not re-enabled by restarting the server. A disabled port does not accept requests until you explicitly enable it.
Port
Specifies the number of the port to which you want to assign a diagnostic listener. Type a port number that is not used by another port on your server.
Alias
An alias for the port. An alias must include one or more of the following: ASCII characters, numbers, underscore (_), period (.), and hyphen (-).
Description
A description of the port.
Package Name
Specifies the package that the server associates with this port. The server always associates the diagnostic port with the WmRoot package. You cannot change the package association of the diagnostic port.
Bind Address
Optional. IP address to which to bind this port. Specify a bind address if your machine has multiple IP addresses and you want the port to use this specific address. If you do not specify a bind address, the server picks one for you.
Backlog
The number of requests that can remain in the queue for an enabled port before Integration Server begins rejecting requests. The default is 200. The maximum value is 65535.
Note:  
This parameter does not apply to disabled ports. Integration Server refuses requests sent to disabled ports.
Keep Alive Timeout
When to close the connection if the server has not received a request from the client within this timeout value (in milliseconds); or when to close the connection if the client has explicitly placed a close request with the server.
Private Threadpool Configuration
Threadpool
Whether the listener will use this pool exclusively for dispatching requests. The existing Integration Server threadpool is a global threadpool. If there is a very high load on this resource, the user may have to wait for the global threadpool to process his request. However, with the private threadpool option enabled, requests coming into this port will not have to compete with other server functions for threads.
Click Enable if you wish to enable the private threadpool settings, or click Disable to disable them. When these settings are enabled, you can change or accept their default values as follows:
Option
Description
Threadpool Min
The minimum number of threads for this private threadpool. The default is 1.
Threadpool Max
The maximum number of threads for this private threadpool. The default is 5.
Threadpool Priority
The Java thread priority.
Important!  
Use this setting with extreme care because it will affect server performance and throughput.
When you view the port’s details, the server reports the total number of private threadpool threads currently in use for the port.
Security
Client Authentication
Specifies the way in which you want the server to authenticate clients that submit requests through this port. Select one of the following:
Option
Description
Username/Password
The server will not request client certificates. Instead it will authenticate clients by their user IDs and passwords.
Digest
Integration Server uses password digest for authentication of all requests. If the client does not provide the authentication information, Integration Server returns an HTTP WWW-Authenticate header with digest scheme to the client requesting for authentication information. If the client provides the required authentication information, Integration Server verifies and validates the request.
Note:  
A port that is configured to use password digest for authentication of client requests will process a request from a user only if the user is configured to allow password digest for authentication.
Request Client Certificates
The server will request client certificates for all requests that come in on this HTTPS port. If the client does not present a certificate, the request proceeds anyway, prompting the user for a user ID and password.
Require Client Certificates
The server will require client certificates for all requests that come in on this HTTPS port. If a client does not supply a client certificate, or if the certificate is not trusted, the request fails.
Use JSSE
If this port should support TLS 1.1 or TLS 1.2, click Yes to create the port using the Java Secure Socket Extension (JSSE) socket factory. The default is No.
Note:  
If you set this value to No, the port supports only SSL 3.0 and TLS 1.0.
Listener Specific Credentials (Optional)
Optional. Use this section to specify the digital certificates that you want the server to use when processing inbound requests over this port. Specify these settings only when you want to use a different set of credentials from the ones specified on the Certificates Screen.
Keystore Alias
Specifies a text identifier for the keystore, which contains private keys and certificates for an Integration Server, partner application, or Integration Server component.
Key Alias
Specifies a text identifier for the private key associated with the keystore alias.
Truststore Alias
Specifies a text identifier for the truststore, which contains the trusted root of a certificate or signing authority (CA).
Save Changes
Updates the server with the new or changed port settings. Your changes take effect immediately.
